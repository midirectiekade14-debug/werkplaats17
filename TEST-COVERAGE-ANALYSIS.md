# Test Coverage Analysis — Werkplaats17

## Executive Summary

**Current test coverage: 0%.** The codebase has no test files, no testing framework, no CI/CD pipeline, and no package manager configuration. The entire application (~4,000 lines across 7 files) is completely untested.

This document identifies the highest-value areas where adding tests would reduce risk and improve confidence in the codebase.

## Priority 1 — Critical Business Logic (High Risk)

### 1. Authentication (`admin.html:486-495`)

The admin panel is protected by a client-side SHA-256 password check. Functions to test:

- **`sha256(m)`** — Hashes input and compares against `PASS_HASH`. A bug here could lock out admins or bypass auth entirely.
- **Auth gate flow** — Correct password sets `sessionStorage`, incorrect password shows error and clears input.

**Suggested tests:**
- `sha256()` produces correct hex digest for known inputs
- Auth gate opens on correct password
- Auth gate rejects incorrect password and shows error state
- Session persistence: re-entering page with `sessionStorage['w17auth'] === 'ok'` skips the gate

### 2. GitHub Publishing (`admin.html:934-978`)

All admin changes are persisted via the GitHub Contents API. Two functions handle this:

- **`publishFile(filepath, content, commitMsg, btn)`** — Encodes content to base64, fetches existing SHA for updates, and PUTs to the API.
- **`publishBinary(filepath, base64data, commitMsg)`** — Same flow for binary files (photos).

**Suggested tests:**
- Correct base64 encoding of UTF-8 content (line 941: `btoa(unescape(encodeURIComponent(content)))`)
- SHA retrieval for file updates vs. new file creation
- Error handling: missing token triggers token modal
- Error handling: API failure shows error toast and resets button state
- Return values: `true` on success, `false` on failure

### 3. Content Editor Data Loading (`admin.html:1095-1118`)

- **`ghDecode(b64)`** — Strips whitespace from GitHub's base64 response before decoding. A subtle bug here corrupts all content.
- **`ceLoadFromGitHub()`** — Fetches `content.js` from GitHub, evaluates it with `new Function()`, and populates the editor.

**Suggested tests:**
- `ghDecode()` correctly handles base64 with embedded newlines
- `ghDecode()` correctly decodes UTF-8 characters (Dutch text with diacritics)
- Fallback to local `CONTENT` object when GitHub is unavailable
- Error handling when returned content is malformed

## Priority 2 — Data Integrity (Medium-High Risk)

### 4. Object Serialization Round-trip (`admin.html:908-926`)

- **`serializeObjects()`** — Reads DOM elements with class `.pl` and extracts their properties into an array of plain objects.
- **`loadObjects(objs)`** — Takes that array and recreates the DOM elements.
- **`generateJS(objs)`** — Converts the array into a valid JavaScript source string for `placed-objects.js`.

**Suggested tests:**
- Round-trip: `loadObjects(serializeObjects())` produces identical state
- `generateJS()` produces syntactically valid JavaScript
- `generateJS()` output can be evaluated back into the original array
- Edge cases: objects with empty strings, zero coordinates, missing optional fields (`rot`, `status`)

### 5. Floorplan Coordinate Math (`admin.html:500-521`, `index.html`)

Core drawing primitives that convert millimeter measurements to pixel positions:

- **`px(mm)`** — Scales mm to pixels (`mm * 0.03`)
- **`mx(x, w)`** — Mirror calculation for right-side units
- **`room()`, `wall()`, `unitLbl()`** — Create positioned DOM elements
- **`drawUnitL()`, `drawUnitM()`** — Draw complete unit layouts

**Suggested tests:**
- `px()` correctly converts known mm values
- `mx()` mirrors coordinates correctly: `mx(x, w) = UW - x - w`
- Snap-to-grid calculations use `SNAP=5` correctly
- Area calculations in room labels are accurate (e.g., `Math.round(kW/1000 * kH/1000)` + 'm²')

### 6. HTML Field Encoding (`admin.html:1153-1179`)

The content editor converts between newlines and `<br>` tags:

- Textarea → data: `value.replace(/\n/g, '<br>')` (line 1165, 1177)
- Data → textarea: `val.replace(/<br\s*\/?>/gi, '\n')` (line 1164)

**Suggested tests:**
- Round-trip: newlines survive encode → decode
- Self-closing `<br/>` and `<br />` variants are handled
- HTML tags within content (bold, italic) are preserved
- `ceWrapSelection()` correctly wraps/unwraps tags (line 1138-1151)

## Priority 3 — UI Interactions (Medium Risk)

### 7. Pan/Zoom Transform System (`admin.html`)

The floorplan editor supports mouse drag (pan), scroll wheel (zoom), and touch gestures:

- **`applyTransform()`** — Applies CSS transform based on `panX`, `panY`, `zoom` state
- **`fitToView()`** — Auto-scales floorplan to fit the viewport
- **Touch gestures** — 1-finger pan, 2-finger pinch zoom

**Suggested tests:**
- Zoom stays within bounds
- Pan offsets update correctly after drag
- `fitToView()` calculates correct scale for different viewport sizes
- Pinch zoom calculates correct scale from touch distance

### 8. Object Drag-and-Drop (`admin.html:595-652`)

- **`place(x, y, opts)`** — Creates a positioned DOM element on the floorplan
- **`selectObj(el)`** — Highlights and opens properties panel
- **Drag handling** — Mousedown/mousemove/mouseup with snap-to-grid

**Suggested tests:**
- Objects snap to grid based on `SNAP` constant
- Objects constrain to floorplan bounds
- Selection/deselection updates properties panel
- Floor switching hides/shows correct objects

## Recommended Testing Setup

Since this is a vanilla JS project with no build system, I recommend:

1. **Initialize npm:** `npm init -y`
2. **Testing framework:** [Vitest](https://vitest.dev/) — fast, modern, zero-config
3. **DOM testing:** [jsdom](https://github.com/jsdom/jsdom) (built into Vitest) or [happy-dom](https://github.com/nicedoc/happy-dom)
4. **E2E testing:** [Playwright](https://playwright.dev/) — for full browser interaction tests (pan/zoom, drag-drop)
5. **API mocking:** [Mock Service Worker (msw)](https://mswjs.io/) — for GitHub API tests

### Extracting testable functions

The biggest prerequisite for unit testing is **extracting pure functions** from the inline `<script>` blocks into importable modules. For example:

```js
// lib/crypto.js
export async function sha256(m) {
  const b = await crypto.subtle.digest('SHA-256', new TextEncoder().encode(m));
  return Array.from(new Uint8Array(b)).map(x => x.toString(16).padStart(2, '0')).join('');
}

// lib/encoding.js
export function ghDecode(b64) {
  return decodeURIComponent(escape(atob(b64.replace(/\s/g, ''))));
}

// lib/coordinates.js
export const S = 0.03;
export const SNAP = 5;
export function px(mm) { return mm * S; }
export function mx(x, w) { return 11500 - x - w; }
```

This modularization would make unit testing straightforward without requiring a full browser environment for most tests.
